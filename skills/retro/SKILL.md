# Retro — 하네스 회고·진화

하네스는 진화하는 시스템이다. 이 스킬은 실행의 흔적(산출물)에서 학습을 추출해 하네스 정의를 개선한다. **기억이 아니라 파일이 근거다** — 대화의 인상이 아니라 워크로그·리포트에 남은 사실로 판단해야 같은 분석을 다음 세션에서도 재현할 수 있다.

## 절대 원칙

1. **제안 → 승인 → 적용.** 하네스 정의(에이전트·스킬·오케스트레이터)는 사용자의 시스템이다. 개선안은 근거와 함께 보고하고, 사용자가 승인한 항목만 적용한다. 자동 적용 금지.
2. **모든 개선안에 근거 산출물을 병기한다.** "어떤 워크로그/리포트의 어떤 기록에서 나왔는가"를 답할 수 없는 개선안은 제안하지 않는다 — 근거 없는 회고는 추측이다.
3. git 작업 금지(사용자 전담)·시크릿 읽기·기록 금지는 여기서도 동일하다.

## 워크플로우

### 1. 입력 수집

- `docs/worklog/` 워크로그, `docs/**/qa-report.md` 등 QA 산출물, `_workspace/` 중간 산출물
- `docs/retro/` 이전 회고 — **반복 문제·재발 추적의 기준선**
- `.claude/agents/`, `.claude/skills/`, CLAUDE.md 변경 이력 — 현재 하네스 정의
- 워크로그가 0건이면 회고를 중단하고 사용자에게 알린다.

### 2. 분석

- **잘된 점** — 유지·강화할 패턴. 근거 산출물 병기.
- **반복 문제** — 같은 유형의 실패·마찰 2회 이상. 이전 회고 지적 사항의 재발 여부를 반드시 대조한다 — 재발은 지난 개선이 불충분했다는 신호이므로 우선순위를 올린다.
- **원인 → 수정 대상 매핑** (harness 스킬 Phase 4와 동일한 경로):

| 증상 | 수정 대상 |
|------|----------|
| 결과물 품질 문제 | 해당 스킬 |
| 역할 혼선·범위 이탈 | 에이전트 정의 |
| 순서·조율 문제 | 오케스트레이터 |
| 트리거 누락·오발동 | description |

### 3. 회고 문서 기록

docs 스킬의 회고 템플릿으로 `docs/retro/{YYYY-MM-DD}-{slug}.md`에 기록한다. 템플릿은 프로젝트 `docs/templates/retro.md`가 단일 출처이고, 없으면 docs 스킬의 절차로 복사한다. 개선안 표의 상태는 전부 "제안"으로 시작한다.

### 4. 승인과 적용

1. 개선안을 근거·위험도와 함께 사용자에게 보고하고 **항목별로** 승인을 받는다.
2. 승인된 항목만 적용한다. 적용마다 CLAUDE.md 변경 이력에 기록한다.
3. harness 스킬의 `scripts/validateHarness.mjs`로 재검증한다 (error 0건).
4. 회고 문서의 개선안 상태와 "4. 적용 결과" 섹션을 갱신한다. 보류 항목은 상태 "보류"로 남긴다 — 다음 회고의 입력이 된다.

## 하네스 연동

guksu-harness가 생성하는 오케스트레이터는 종료 절차에서 회고를 제안한다(강요하지 않음). 사용자가 동의하면 이 스킬이 실행된다. 독립적으로도 언제든 호출 가능하다.
