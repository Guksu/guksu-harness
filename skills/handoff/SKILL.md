# Handoff — 세션 인계

작업은 세션보다 오래 산다. 이 스킬은 진행 중인 작업의 상태를 파일로 옮겨, 컨텍스트가 사라져도 다음 세션(또는 다른 에이전트)이 이어받게 한다. **독자는 이 대화를 전혀 보지 못한 새 세션이다** — 대화 맥락 없이 읽혀야 인계다.

## 두 가지 모드

### 작성·갱신 (세션을 넘길 때)

1. **템플릿 확정** — 프로젝트 `docs/templates/handoff.md`가 단일 출처. 없으면 docs 스킬의 절차로 `assets/templates/handoff.md`를 복사한다.
2. **기존 문서 확인** — `docs/handoff/`에 같은 작업 흐름의 문서가 있으면 새로 만들지 않고 갱신한다. **작업 흐름(work stream)당 1개 문서**다 — 인계 문서가 여러 개면 다음 세션이 어느 것이 최신인지 알 수 없다.
3. **경로** — `docs/handoff/{slug}.md`. slug는 작업 흐름 이름(날짜 아님 — 갱신형 문서라 날짜가 붙으면 파일이 늘어난다).
4. **채우기 규칙**:
   - "2. 진행 상황"의 모든 항목에 근거(파일 경로·산출물)를 병기한다 — 다음 세션은 주장 대신 파일을 확인한다.
   - "3. 시도와 결과"는 **누적 기록하며 지우지 않는다** — 같은 막다른 길을 반복하지 않는 것이 이 섹션의 존재 이유다.
   - "4. 다음 단계"의 첫 항목은 "무엇을 읽고 시작하라"부터 — 이어받는 쪽의 첫 행동을 결정해 준다.
   - 시크릿은 기록하지 않는다. 대화에만 있고 파일에 없는 결정 사항이 있으면 지금 파일로 옮긴다 — 인계 후 대화는 사라진다.

### 인수 (이어받을 때)

1. `docs/handoff/`에서 해당 작업 흐름 문서를 읽는다. 어느 것인지 모호하면 최근 갱신 문서를 후보로 사용자에게 확인한다.
2. "2. 진행 상황"의 근거 파일을 실제로 열어 문서와 현실이 일치하는지 검증한다 — 인계 후 파일이 바뀌었을 수 있다. 불일치하면 추측하지 말고 문서를 현실에 맞게 갱신한 뒤 진행한다.
3. "3. 시도와 결과"를 읽고 실패한 접근을 반복하지 않는다.
4. "4. 다음 단계"부터 착수하고, 의미 있는 진전마다 문서를 갱신한다.

## 종료

작업 흐름이 완료되면 인계 문서의 상태를 "완료"로 바꾸고, 최종 결과는 docs 스킬의 워크로그로 기록한다 — 인계 문서는 "진행 중 상태"의 스냅샷이고, 워크로그가 완결된 기록이다. 완료된 인계 문서는 삭제하지 않는다(감사 추적).

## 하네스 연동

guksu-harness가 생성하는 하네스에서는 세션 중단·컨텍스트 한계 상황의 대응으로 이 스킬이 오케스트레이터 에러 핸들링에 명시된다. 독립적으로도 언제든 호출 가능하다.
