# 실행 모드 — Workflow · 에이전트 팀 · 서브 에이전트

하네스의 실행 모드를 결정하는 기준과 모드별 사용법. 핵심 판별 질문은 하나다:

> **"이 작업의 제어 흐름을 지금 코드로 쓸 수 있는가?"**

- 쓸 수 있다(팬아웃 대상이 정해져 있고, 단계가 고정이고, 분기 조건이 명확) → **Workflow**
- 쓸 수 없다(진행하면서 협상·재작업·계약 조정이 일어난다) → **에이전트 팀**
- 흐름이랄 게 없다(전문가 한 명에게 맡기고 결과만 받으면 끝) → **서브 에이전트**

## 1. 결정 트리

```
협업자가 2명 이상인가?
├─ 아니오 → 서브 에이전트 (Agent 도구, 필요 시 run_in_background)
└─ 예 → 제어 흐름을 미리 코드로 쓸 수 있는가?
    ├─ 예 (팬아웃/파이프라인/반복, 항목 목록이 결정 가능)
    │   → Workflow  ※ 항목 수십~수백 개, 스키마 검증 산출물, 토큰 버짓 관리에 특히 유리
    └─ 아니오 (진행 중 피드백 루프·계약 협상·재작업이 본질)
        → 에이전트 팀  ※ 기획↔구현↔QA 같은 생성-검증 루프가 대표 사례
```

주의: "여러 에이전트가 협업한다"는 사실만으로 에이전트 팀을 고르지 않는다. 협업의 형태가 **사전에 결정 가능한 데이터 흐름**이면 Workflow가 더 싸고 빠르고 재현 가능하다. 팀은 통신 오버헤드가 실제 가치(토론·계약 조정)를 만들 때만 선택한다.

## 2. Workflow — 결정적 오케스트레이션

`Workflow` 도구는 자바스크립트 스크립트로 서브 에이전트들을 결정적으로 조율한다. 모델이 "다음에 누굴 부를까"를 추론하지 않으므로 재현 가능하고, 수십~수백 항목 팬아웃에서 누락이 없다.

**언제 선택하는가:**
- 항목 목록 × 동일 처리: 파일 50개 마이그레이션, 챕터 20개 검수, 소스 30건 리서치
- 다단계 파이프라인: 찾기 → 검증 → 종합 (단계 간 의존이 데이터로 표현됨)
- 산출물 스키마 강제: `schema` 옵션으로 JSON Schema 검증된 구조화 출력을 받아야 할 때
- 적대적 검증: 발견 1건당 반박 에이전트 N명 투표 같은 품질 패턴

**핵심 API (스크립트 안에서 사용):**

| 함수 | 역할 |
|------|------|
| `agent(prompt, opts)` | 서브 에이전트 1개 스폰. `opts.schema`(JSON Schema)로 구조화 출력 강제, `opts.agentType`으로 `.claude/agents/`의 커스텀 에이전트 사용, `opts.isolation: 'worktree'`로 병렬 파일 수정 격리 |
| `pipeline(items, ...stages)` | 항목별로 단계를 독립 진행(단계 간 배리어 없음). **다단계 작업의 기본값** |
| `parallel(thunks)` | 전부 끝날 때까지 대기(배리어). 전체 결과가 동시에 필요할 때만(중복 제거, 0건 조기 종료) |
| `phase(title)` / `log(msg)` | 진행 상황 그룹핑/표시 |
| `budget` | 토큰 버짓 조회 — 버짓 기반 반복 깊이 조절 |

**스크립트 골격 예시 (찾기→검증 파이프라인):**

```js
export const meta = {
  name: 'review-chapters',
  description: '챕터별 검수 후 발견사항 적대적 검증',
  phases: [{ title: '검수' }, { title: '검증' }],
}
const results = await pipeline(
  args.chapters,
  (ch) => agent(`${ch} 검수...`, { phase: '검수', schema: FINDINGS_SCHEMA }),
  (review) => parallel(review.findings.map((f) => () =>
    agent(`반박 시도: ${f.title}`, { phase: '검증', schema: VERDICT_SCHEMA })
      .then((v) => ({ ...f, verdict: v })))),
)
return results.flat().filter(Boolean).filter((f) => f.verdict?.isReal)
```

**주의사항:**
- 스크립트는 순수 JS(TS 문법 불가), `Date.now()`/`Math.random()` 사용 불가(재개 기능이 깨짐) — 타임스탬프는 `args`로 주입
- `meta`는 순수 리터럴이어야 한다
- 모델 오버라이드(`opts.model`)는 기본 생략 — 세션 모델 상속이 원칙
- Workflow는 사용자가 멀티 에이전트 오케스트레이션을 명시적으로 요청한 맥락에서 실행된다. 하네스의 오케스트레이터 스킬이 Workflow를 실행 수단으로 쓰는 경우, 그 사실을 스킬 description과 본문에 명시해 사용자가 스킬 호출로 이미 옵트인했음을 분명히 한다
- 에이전트 정의 파일(`.claude/agents/`)은 Workflow 모드에서도 만든다 — `agentType` 옵션으로 연결

## 3. 에이전트 팀

`TeamCreate`로 팀을 만들고 `TaskCreate`로 의존성 있는 작업을 등록하면, 팀원들이 `SendMessage`로 직접 통신하며 자체 조율한다. 리더(오케스트레이터)는 진행을 모니터링하고 결과를 종합한다.

**언제 선택하는가:**
- 생성-검증 피드백 루프: developer가 모듈을 완성할 때마다 QA가 교차검증하고 수정 요청을 보내는 구조
- 계약 협상: planner의 계획이 모호하면 developer가 질문하고 계획이 갱신되는 구조
- 중간 산출물 상호 참조: 서로의 진행 중 결과를 보고 방향을 조정해야 할 때

**골격:**

```
TeamCreate(team_name, members: [{ name, agent_type, prompt }...])
TaskCreate(tasks: [{ title, assignee, depends_on }...])
→ 팀원 자체 조율 (SendMessage)
→ 리더: TaskGet으로 모니터링, 막힌 팀원 지원
→ 산출물 파일 확인 → 종료 판정 → TeamDelete
```

**팀 크기:** 3명의 집중된 팀원이 5명의 산만한 팀원보다 낫다.

| 작업 규모 | 권장 팀원 수 |
|----------|------------|
| 소규모 (작업 5~10개) | 2~3명 |
| 중규모 (10~20개) | 3~5명 |
| 대규모 (20개+) | 5~7명 |

**제약:** 팀은 세션당 하나만 활성화된다. Phase별로 다른 전문가 조합이 필요하면 산출물을 파일로 저장 → `TeamDelete` → 새 팀 생성.

## 4. 서브 에이전트

`Agent` 도구로 직접 호출하고 결과만 회수한다. 병렬 실행은 `run_in_background: true` 또는 한 메시지에 여러 호출.

**언제 선택하는가:**
- 단일 전문가 작업(탐색·리뷰·문서 생성)으로 충분할 때
- 팀 통신이 구조적으로 불필요(결과 전달만)하고 팀 오버헤드가 이득보다 클 때

**빌트인 타입 활용:** 읽기 전용 탐색은 `Explore`, 계획 수립은 `Plan`, 실행이 필요하면 `general-purpose`. 커스텀 정의(`.claude/agents/{name}.md`)가 있으면 `subagent_type`으로 지정한다.

## 5. 하이브리드

Phase마다 특성이 다르면 모드를 섞는다. 오케스트레이터의 각 Phase 상단에 실행 모드를 명시한다 (예: `**실행 모드:** Workflow`).

자주 쓰는 조합:
- **Workflow 수집 → 팀 통합**: 대량 자료를 Workflow로 병렬 수집·검증 → 팀이 토론·합의로 통합
- **팀 생성 → Workflow 검증**: 팀이 기능을 구현 → Workflow가 파일별 검수를 결정적으로 팬아웃
- **서브 정찰 → Workflow 본대**: 서브 에이전트로 작업 목록(work-list)을 먼저 발견 → 목록을 Workflow에 `args`로 전달

## 6. 아키텍처 패턴 → 모드 매핑

| 패턴 | 구조 | 기본 모드 |
|------|------|----------|
| 파이프라인 | A→B→C 순차 의존 | 단계가 고정이면 Workflow `pipeline`, 단계 간 협상이 있으면 팀 |
| 팬아웃/팬인 | 병렬 독립 작업 후 종합 | Workflow (`parallel`/`pipeline`) |
| 전문가 풀 | 상황별 선택 호출 | 서브 에이전트 |
| 생성-검증 | 생성 후 품질 검수 루프 | 재작업 루프가 본질이면 팀, 1회 검증이면 Workflow |
| 감독자 | 중앙이 상태 관리·동적 분배 | 에이전트 팀 (리더 = 감독자) |
| 계층적 위임 | 상위가 하위에 재귀 위임 | 하이브리드 (상위 팀 + 하위 Workflow/서브) |
