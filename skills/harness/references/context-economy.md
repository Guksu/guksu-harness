# 컨텍스트 경제 — 토큰 절약 설계 규칙

절대 규칙 7("컨텍스트는 절약형으로 설계한다")의 상세. 하네스가 소비하는 토큰의 대부분은 **매 턴 반복 지불하는 상시 로딩**과 **한 컨텍스트에 밀어 넣은 대량 읽기**에서 나온다. 이 두 가지를 설계 단계에서 통제하면 같은 하네스가 몇 분의 일 비용으로 돈다.

## 1. 플랫폼이 이미 해주는 것 — 재구현 금지

Claude Code가 자동으로 수행하므로 하네스가 스킬·훅으로 다시 만들지 않는다. 중복 구현은 토큰을 아끼지 못하고 오히려 상시 로딩만 늘린다.

| 자동 메커니즘 | 동작 | 설계 시사점 |
|--------------|------|------------|
| **프롬프트 캐시** | 시스템 프롬프트 → CLAUDE.md/프로젝트 컨텍스트 → 대화 순의 캐시 레이어. 바뀌지 않은 앞부분은 표준 입력 단가의 ~10%로 재사용되고, 캐시 히트마다 TTL이 리셋된다 | "CLAUDE.md를 다시 안 읽게 캐싱하는 스킬"은 불필요 — 이미 그렇게 동작한다 |
| **CLAUDE.md 세션당 1회 로드** | 세션 시작 시 읽어 컨텍스트에 유지. 매 턴 디스크 재읽기 없음 | 대신 크기가 곧 매 턴 비용이다 — §3의 다이어트 규칙이 실효 지점 |
| **스킬 지연 로딩** | 상시 로딩은 name+description뿐, 본문은 트리거 시에만 | description은 짧고 pushy하게, 무게는 본문·references/로 (skill-authoring.md) |
| **MCP 스키마 지연 로딩** | 툴 이름만 상시(~120토큰/서버), 전체 스키마는 사용 시점에 로드 | 그래도 안 쓰는 MCP 서버는 정리 대상 — 이름 목록도 누적되면 비용이다 |
| **자동 컴팩션** | 컨텍스트 ~80% 도달 시 대화를 요약 | 컴팩션은 손실 압축이다 — 도달 전에 handoff·워크로그로 파일에 옮기는 쪽이 안전하다 |

## 2. 캐시를 깨는 행동

프롬프트 캐시는 "앞부분이 그대로"일 때만 산다. 세션 중 아래 행동은 캐시를 무효화해 그 턴 전체를 비캐시 단가로 만든다 — 오케스트레이터·루프가 **반복 실행 중에** 하지 않도록 설계한다:

- **모델·reasoning effort 전환** — 모델 정책 "세션 상속 기본"(agent-design.md §4)은 캐시 관점에서도 옳다. 루프 중간에 모델을 갈아타는 설계는 매 전환마다 캐시 재구축 비용을 낸다.
- **MCP 서버 연결/해제, 플러그인 활성/비활성** — 필요한 연결은 세션 시작 시점에 끝내둔다.
- **컴팩션** — 자동이든 수동(/compact)이든 그 턴은 비싸다. §1의 시사점대로 파일 이관으로 예방한다.

반대로 **파일 편집·스킬 호출·권한 모드 전환은 캐시를 유지한다** — 루프 본체에서 자유롭게 써도 된다.

## 3. 생성 규칙 — 하네스 산출물의 토큰 규율

원리는 하나다: **상시 비용(매 턴)과 조건부 비용(필요할 때)을 구분하고, 모든 지침·데이터를 가능한 낮은 층으로 내린다.**

| 층 | 로딩 시점 | 무엇을 두나 |
|----|----------|------------|
| CLAUDE.md | 매 턴 (상시) | 포인터·트리거 규칙·변경 이력만 — **~200줄 이내** |
| `.claude/rules/*.md` + `paths:` | 매칭 파일을 열 때만 | 특정 파일 유형·디렉토리에만 적용되는 컨벤션 |
| 스킬 본문 | 트리거 시 | 작업 절차 (≤500줄) |
| references/ | 스킬이 지시할 때 | 심화 가이드 |
| scripts/ | 로딩 없이 실행 | 결정적 반복 작업 |

개별 규칙:

1. **CLAUDE.md는 포인터만, ~200줄 이내.** Phase 4 등록 형식(트리거 + 변경 이력)을 유지하고 에이전트/스킬 목록·긴 컨벤션을 넣지 않는다 — 파일 시스템과 오케스트레이터가 단일 출처다. 변경 이력 테이블이 길어지면 오래된 행을 워크로그로 이관하고 최근 이력만 남긴다.
2. **조건부 지침은 `.claude/rules/`로 내린다.** frontmatter의 `paths:` glob에 매칭되는 파일을 열 때만 로드된다. "마이그레이션 파일은 이렇게", "테스트는 이 컨벤션으로" 같은 지침이 CLAUDE.md에 있으면 관련 없는 턴에도 매번 과금된다:

   ```markdown
   ---
   paths:
     - "supabase/migrations/**"
   ---
   마이그레이션 작성 규칙: ...
   ```
3. **대량 읽기는 서브 에이전트로 격리한다.** 리서치·다중 파일 탐색·로그 분석은 메인 컨텍스트에서 직접 읽지 않고 서브 에이전트에 위임해 요약만 회수한다(agent-design.md 분리 기준의 "컨텍스트" 축). 메인 대화에 들어온 토큰은 이후 모든 턴이 다시 지불한다 — 대화가 길수록 메인 컨텍스트 1토큰이 가장 비싼 토큰이다.
4. **대형 출력은 스크립트·훅으로 전처리한다.** 수천 줄 로그·데이터를 통째로 읽히지 말고 grep·jq 등으로 필요한 부분만 추출해 전달한다. 같은 전처리가 반복되면 스킬 scripts/로 번들링한다(skill-authoring.md "반복 코드는 번들링").
5. **반복 소비되는 분석 결과는 파일로 캐싱한다.** 프롬프트 캐시는 세션 안에서만 산다 — 세션이 바뀌면 같은 파일을 처음부터 다시 읽는다. 대형 모듈의 분석 요약은 `digest` 스킬(내용 해시로 신선도를 검증하는 `docs/digests/` 캐시)로 남기고 다음 세션이 원문 대신 다이제스트를 먼저 소비하게 한다. 작업 상태는 handoff, 완결 기록은 워크로그가 담당한다.

## 4. 측정과 진단

| 수단 | 무엇을 보나 |
|------|------------|
| `/context` | 현재 컨텍스트 구성(시스템·CLAUDE.md·대화별 점유) — 상시 로딩 비대 진단 |
| `/usage` | 스킬·서브 에이전트·MCP 서버별 사용량 비중 — 비용 주범 식별 |
| 검증자 게이트 `maxTokens` | 루프의 토큰 예산 강제 — 초과 시 자동 중단·보고 (hooks-and-permissions.md §4) |

하네스 점검(Phase 0 운영/유지보수)에서 CLAUDE.md 줄 수와 description 총량이 계속 늘고 있으면 회고(retro) 대상이다 — 상시 로딩은 한 번 늘면 매 턴 과금된다.
