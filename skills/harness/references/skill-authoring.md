## 1. 스킬 구조

```
skill-name/
├── SKILL.md          (필수: YAML frontmatter + 본문)
├── scripts/          (선택: 반복·결정적 작업용 실행 코드 — 로딩 없이 실행 가능)
├── references/       (선택: 조건부 로딩하는 참조 문서)
└── assets/           (선택: 출력에 쓰이는 템플릿·이미지)
```

frontmatter의 `name`은 디렉토리명과 일치시킨다 (`validateHarness.mjs`가 검사한다).

## 2. Description — 유일한 트리거 메커니즘

description은 항상 컨텍스트에 존재하는 유일한 부분이며, 스킬이 트리거될지를 전적으로 결정한다. Claude는 트리거를 보수적으로 판단하므로 **적극적(pushy)으로** 작성한다.

**나쁜 예:** `"PDF 문서를 처리하는 스킬"`
**좋은 예:** `"PDF 읽기, 텍스트/테이블 추출, 병합, 분할, 암호화 등 모든 PDF 작업을 수행. .pdf 파일을 언급하거나 PDF 산출물을 요청하면 반드시 이 스킬을 사용할 것."`

**포함할 것:**
1. 스킬이 하는 일 (구체적 동사로 나열)
2. 트리거 상황 — 사용자가 실제로 입력할 표현들 (공식적/캐주얼 모두)
3. **후속 작업 키워드** — "다시 실행", "재실행", "업데이트", "수정", "보완", "{부분}만 다시", "이전 결과 개선". 초기 생성 키워드만으로는 후속 요청이 트리거되지 않는다.
4. 유사하지만 트리거하면 안 되는 경우와의 구분 ("단순 질문은 직접 응답 가능" 등)

## 3. 본문 작성 원칙

| 원칙 | 설명 |
|------|------|
| **Why를 설명하라** | "ALWAYS/NEVER" 강압 대신 이유를 전달한다. 이유를 이해한 모델은 명시되지 않은 엣지 케이스에서도 올바르게 판단한다. |
| **Lean하게 유지** | 컨텍스트는 공공재다. 무게를 벌지 않는 문장은 삭제하거나 references/로 옮긴다. |
| **일반화하라** | 특정 예시에만 맞는 좁은 규칙(오버피팅) 대신 원리를 쓴다. 피드백을 반영할 때도 그 사례만 고치지 말고 일반 원칙으로 승격시킨다. |
| **반복 코드는 번들링** | 에이전트들이 매번 같은 스크립트를 작성하는 패턴이 보이면 `scripts/`에 미리 번들링한다 (결정적·재사용·컨텍스트 절약). |
| **명령형으로 작성** | "~한다", "~하라" 어조. 설명문이 아니라 작업 지시서다. |

## 4. Progressive Disclosure

3단계 로딩으로 컨텍스트 비용을 통제한다:

| 단계 | 로딩 시점 | 크기 목표 |
|------|----------|----------|
| Metadata (name+description) | 항상 | ~100단어 |
| SKILL.md 본문 | 스킬 트리거 시 | ≤500줄 |
| references/ · scripts/ | 필요할 때만 (scripts는 로딩 없이 실행) | 제한 없음 |

**크기 관리 규칙:**
- SKILL.md가 500줄에 근접하면 세부를 references/로 분리하고, 본문에는 "언제 이 파일을 읽으라"는 포인터를 남긴다.
- 300줄 이상의 reference에는 상단에 목차를 둔다.
- 도메인/플랫폼별 변형은 references/ 하위에 파일을 나눠 관련 파일만 로드되게 한다 (예: `references/aws.md`는 AWS 선택 시에만).

## 5. 스킬-에이전트 연결

- 에이전트 1개 ↔ 스킬 1~N개. 여러 에이전트가 한 스킬을 공유해도 된다.
- 스킬은 "어떻게", 에이전트는 "누가"를 담는다. 에이전트 정의에 작업 절차가 길게 들어가기 시작하면 스킬로 분리할 신호다.
- 에이전트 정의에는 "작업 전 `{skill}` 스킬을 읽어라(또는 Skill 도구로 호출하라)"를 명시해 연결을 끊기지 않게 한다.
