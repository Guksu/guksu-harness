# 오케스트레이터 템플릿 — 모드별 골격 · 데이터 전달 · 에러 핸들링

오케스트레이터는 스킬의 특수한 형태로, "누가 언제 어떤 순서로 협업하는가"를 정의한다. 개별 스킬이 "어떻게"라면 오케스트레이터는 흐름이다. **기존 확장 시에는 새로 만들지 않고 기존 오케스트레이터를 수정한다.**

## 1. 공통 골격

```markdown
---
name: {domain}-orchestrator
description: "{도메인} 작업을 조율하는 오케스트레이터. {트리거 표현들}.
  후속 작업: 재실행, 수정, 보완, {부분}만 다시, 이전 결과 개선 요청 시에도 반드시 이 스킬을 사용."
---

# {도메인} Orchestrator

## 실행 모드: {직접 실행 | 서브 에이전트 | 루프 | Workflow | 에이전트 팀 | 하이브리드(Phase별 명시)}
선택 이유 한 줄 — 사다리(execution-modes §1)에서 더 싼 모드로 안 되는 근거를 적는다.

## 절대 규칙
0. TDD 기본 (코드 생성 시) — 종료 기준에 테스트 전체 통과 포함.
1. {도메인 단위 규칙 — 예: 1 스프린트 = 1 기능}
2. git 작업 절대 금지 — 커밋·푸시는 사용자가 직접. 종료 시 안내만. (예외 ①: 사용자 확인된 브랜치 전환은 branch 스킬이 git switch로 수행. 예외 ②: 사용자가 명시 요청한 커밋·PR 업로드는 pr 스킬이 git-flow(main·dev·feat)로 수행 — 커밋 메시지·PR 본문에 Claude 작성 표기 절대 금지)
3. 산출물은 파일 기반 — 약속된 경로에 쓰고 읽으며, 중간 산출물은 삭제하지 않는다(감사 추적·부분 재실행).
4. 설계 문서가 단일 출처 — 어긋나면 사용자에게 확인.
5. QA는 경계면 교차검증 + incremental — 생산자↔소비자 shape 비교, 모듈 완성 직후마다.
6. 시크릿 읽기·기록 금지 — `.env`·credential을 읽지 않고 산출물에 토큰/키를 남기지 않는다.
7. 컨텍스트 경제 — 대량 읽기·리서치는 서브 에이전트로 격리해 요약만 회수, 대형 출력은 스크립트로 전처리. 반복 실행 중 모델·effort 전환 금지(캐시 무효화).

## 에이전트 구성
| 팀원 | 타입 | 역할 | 스킬 | 출력 |
|------|------|------|------|------|

## 워크플로우
### Phase 0: 컨텍스트 확인  ← 필수 (아래 §2)
### Phase 1~N: {도메인 흐름}

###       → 채팅에는 요약·보고서 경로·검토 항목 수만 + "커밋은 직접 진행하세요" 안내 + 회고 제안(retro 스킬 — 강요하지 않음)

## 데이터 흐름
(다이어그램 — 누가 무엇을 쓰고 누가 읽는가)

(표 — 상황별 전략)

```

## 2. Phase 0: 컨텍스트 확인 — 후속 작업 지원

오케스트레이터는 초기 실행뿐 아니라 후속 작업도 처리해야 한다.

**첫 단계는 작업 브랜치 확인이다** — 파일 변경이 예정된 실행이면 산출물 확인 전에 `branch` 스킬로 현재 브랜치를 점검한다. 보호 브랜치(main 등) 위라면 사용자 확인을 받아 작업 브랜치로 전환하고 시작한다(branchGuard 훅이 어차피 편집을 차단하므로, 늦게 발견할수록 낭비가 크다).

이어서 기존 산출물을 확인해 실행 모드를 결정한다:

| 상태 | 판정 | 동작 |
|------|------|------|
| 산출물 폴더 없음 | 초기 실행 | 전체 워크플로우 |
| 산출물 존재 + 부분 수정 요청 | **부분 재실행** | 해당 에이전트만 재호출 (기존 산출물을 입력으로 전달). Workflow 모드면 `resumeFromRunId`로 캐시 재개 |
| 산출물 존재 + 새 입력 제공 | 새 실행 | 기존 산출물을 `_prev/`로 이동 후 전체 실행 |

에이전트 정의에도 재호출 지침을 포함시킨다: "이전 산출물이 존재하면 읽고 개선점을 반영, 피드백이 주어지면 해당 부분만 수정."

## 3. 모드별 실행 골격

### 에이전트 팀

세션의 암시적 단일 팀을 쓴다 — 팀 생성·해체 단계가 없다 (`TeamCreate`/`TeamDelete`는 구 API로 제거됨).

```
Agent(name: "planner", subagent_type: "{custom-or-builtin}",
      prompt: "{에이전트 정의 전문 + 이번 작업 컨텍스트 + 산출물 경로}")
Agent(name: "developer", ...)   ← 독립 팀원은 한 메시지에 병렬 스폰
Agent(name: "qa", ...)
TaskCreate(tasks: [
  { title: "기획: ...", assignee: "planner" },
  { title: "구현: ...", assignee: "developer", depends_on: ["기획: ..."] },
  { title: "검증: ...", assignee: "qa", depends_on: ["구현: ..."] },
])
→ 팀원 자체 조율 (SendMessage) — 리더는 TaskList/TaskGet으로 모니터링, 막힌 팀원은 SendMessage로 지원
→ 종료 판정 (산출물 + 테스트 통과 확인) → 사용자 보고
```

### Workflow

스크립트를 `프로젝트/.claude/workflows/{domain}-{task}.mjs`에 저장 워크플로우로 만들고, 오케스트레이터 스킬은 이름으로 호출하며 가변 입력은 `args`로 주입한다. 스크립트가 스킬 본문의 골격 텍스트로만 존재하면 실행마다 재작성되어 결정성이 깨진다.

```
Phase 0에서 작업 목록(work-list)을 확정 → Workflow({name: "{domain}-{task}", args: { items, ... }}) 실행
→ 반환된 구조화 결과를 산출물 파일로 기록 → 사용자 보고
```

- 작업 목록이 미리 결정 불가능하면 서브 에이전트로 먼저 정찰(scout)해 목록을 만든 뒤 Workflow에 넘긴다 (하이브리드).
- 부분 재실행·중단 복구는 `resumeFromRunId`로 처리한다 — 완료된 `agent()` 호출은 캐시로 즉시 반환되므로 수정분만 재실행된다. 이를 위해 실행 후 runId를 산출물(실행 로그)에 기록해 둔다.

### 서브 에이전트

```
Agent(agent-1, run_in_background: true)   ← 독립 작업은 병렬
Agent(agent-2, run_in_background: true)
→ 결과 수집 → 통합 산출물 생성 → 사용자 보고
```

### 루프

설계·실행 절차는 loop 스킬이 전담한다 — 오케스트레이터는 모드 선언과 포인터만 담는다:

```
루프 명세: docs/loops/{slug}.md (4요소 + 안전장치 — 사용자 확인 후 실행)
검증자 게이트: .claude/hooks/verifierGate.config.json — 명세의 검증자·안전장치 값과 일치
한 바퀴 내부 실행: {직접 실행 | 서브 에이전트 | Workflow} (위 골격 재사용)
```

## 4. 데이터 전달 프로토콜

**핸드오프는 명시적 계약이다.** 에이전트 간 인계는 자유 서술 대화가 아니라 **약속된 상태 키**(파일 경로 + 형식)로 넘긴다 — Google ADK가 `output_key → session.state`로 강제하는 것과 같은 원리다. 오케스트레이터는 각 단계의 **생산자·소비자·경로·형식**을 아래 표처럼 고정해 두고, 그 계약을 에이전트 정의에 명시한다. 계약이 흐릿하면 소비자가 생산자의 출력 shape을 추측하게 되고, 그게 경계면 버그의 근원이다(agent-design QA 가이드).

| 전략 | 방식 | 적용 모드 | 적합한 경우 |
|------|------|----------|-----------|
| 파일 기반 | 약속된 경로에 쓰고 읽음 | 전체 | 구조화 산출물, 감사 추적, 세션 간 연속성 — **기본값** |
| 스키마 기반 | Workflow `schema` 옵션 | Workflow | 검증된 구조화 데이터를 코드로 후처리 |
| 태스크 기반 | TaskCreate/TaskUpdate | 팀 | 진행 추적, 의존 관계 |
| 메시지 기반 | SendMessage | 팀 | 실시간 조율, 수정 요청 |
| 반환값 기반 | Agent 반환 메시지 | 서브 | 단발 결과 수집 |

**파일 기반 규칙:**
- 중간 산출물은 `_workspace/` 또는 `docs/{domain}/{slug}/`에 저장, 최종만 사용자 지정 경로로
- 파일명 컨벤션: `{순번}_{agent}_{artifact}.{ext}` 또는 도메인 고정 이름(`plan.md`, `dev-notes.md`, `qa-report.md`)
- 작업 기록은 `docs/templates/worklog.md` 공통 템플릿(1.개요/2.작업내용/3.주의사항)으로 `docs/worklog/{YYYY-MM-DD}-{slug}.md`에 남긴다 — 작업한 에이전트 본인이 기록하고, 병렬 에이전트는 `-{agent}` 접미사로 각자 파일을 만든다 (docs 스킬 참조)
- 중간 산출물은 삭제하지 않는다 (사후 검증·감사 추적·부분 재실행의 입력)

## 5. 에러 핸들링

오케스트레이터에 최소 다음 상황별 전략을 표로 포함한다:

| 상황 | 기본 전략 |
|------|----------|
| 에이전트 1명 실패/중지 | 1회 재시도 → 재실패 시 해당 결과 없이 진행하고 산출물에 누락 명시 |
| 검증 루프 무한 반복 | 2~3회 초과 시 잔여 이슈를 "미해결"로 기록, 사용자에게 판단 요청 |
| 산출물 간 충돌 | 삭제하지 않고 출처 병기, 사용자에게 확인 |
| 작업이 범위보다 큼 | 분할 제안, 첫 조각만 진행 |
| 세션 중단·컨텍스트 한계 임박 | handoff 스킬로 인계 문서(`docs/handoff/{slug}.md`)를 작성·갱신한 뒤 중단 — 다음 세션이 이어받는다 |
| git 명령 시도 | 1차 방어는 PreToolUse 훅(자동 차단, `hooks-and-permissions.md`) — 훅이 없는 환경이면 즉시 중단하고 사용자 전담임을 재고지 |

## 6. 실전 예시 — muklog 스프린트 오케스트레이터

검증된 실전 구성(생성-검증 + 파이프라인, 에이전트 팀 모드)의 요약:

- **팀**: sprint-planner(기획) → developer(구현) → qa-inspector(검증, general-purpose 타입)
- **단위 규칙**: 1 스프린트 = 1 기능. 산출물은 `docs/sprint/sprint-{YYYYMMDD}-{name}/`에 `plan.md` / `dev-notes.md` / `qa-report.md`
- **TDD**: planner가 인수조건을 테스트 케이스로 정의 → developer가 실패 테스트 먼저 작성 → qa가 테스트 존재·의미·통과 검증. 종료 기준 = 테스트 전체 통과 + 타입체크
- **incremental QA**: developer가 모듈 완성마다 qa에게 교차검증 요청(생산자/소비자 경로 명시), qa는 발견 즉시 파일:라인 + 수정 방법으로 회신
- **dev-notes.md에 생산자↔소비자 매핑 명시** — QA가 경계면을 교차검증할 수 있는 입력이 된다
- **종료**: qa-report 전 항목 통과 확인 → HTML 보고서 생성(report 스킬) → 채팅에 요약·경로 + "커밋은 직접 진행하세요" 안내 + 다음 후보 제시 + 회고 제안(retro 스킬)
