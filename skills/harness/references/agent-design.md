## 1. 분리 기준 4축

에이전트를 나눌지 합칠지는 4축으로 판단한다. 2축 이상에서 분리 이득이 있을 때만 나눈다 — 에이전트가 많을수록 조율 비용이 커진다.

| 축 | 분리하는 경우 | 합치는 경우 |
|----|-------------|------------|
| **전문성** | 요구 지식·판단 기준이 서로 다름 (기획 vs 구현 vs 검증) | 같은 지식으로 처리 가능 |
| **병렬성** | 동시에 진행하면 벽시계 시간이 줄어듦 | 순차 의존이라 병렬 이득 없음 |
| **컨텍스트** | 한 컨텍스트에 담기엔 입력이 너무 큼 | 컨텍스트 여유 충분 |
| **재사용성** | 다른 워크플로우에서도 단독 호출됨 | 이 워크플로우 전용 |

**생성자와 검증자는 반드시 분리한다.** 자기 결과물을 자기가 검증하면 같은 맹점을 공유한다.

## 2. 중복·재사용 검토

신규 에이전트/스킬 생성 전 `프로젝트/.claude/agents/`, `.claude/skills/`의 기존 항목과 대조한다. 하네스를 반복 구축하면 역할이 겹치는 에이전트가 다른 이름으로 누적되기 쉽다.

| 판정 | 기준 | 조치 |
|------|------|------|
| 동일 | 역할·입출력이 같음 | 기존 것을 그대로 사용 |
| 포함 | 기존 역할의 부분집합 | 기존 것을 사용, 프롬프트로 범위 한정 |
| 확장 | 기존 역할 + α | 기존 정의를 수정(변경 이력 기록) |
| 신규 | 겹침 없음 | 새로 생성 |

## 3. 정의 파일 템플릿

**모든 에이전트는 `프로젝트/.claude/agents/{name}.md` 파일로 정의한다.** Agent 도구의 prompt에 역할을 직접 넣고 끝내는 것은 금지 — 파일이 있어야 다음 세션에서 재사용되고, 협업 프로토콜이 명시되어야 팀 품질이 보장된다. 빌트인 타입(`general-purpose` 등)을 쓰더라도 정의 파일은 만들고, 호출 시 정의 전문을 프롬프트에 포함한다.

```markdown
---
name: {kebab-case-이름}
description: "{역할 한 줄}. {어떤 작업 시 호출되는지}."
---

# {이름} — {한 줄 정체성}

당신은 {프로젝트}의 {역할}입니다. {전문 영역 요약}.

## 핵심 역할
1~3개. 이 에이전트가 책임지는 결과.

## 작업 원칙
- 절대 규칙 반영: git 작업 금지(사용자 전담), TDD 기본(코드 생성 시), 단일 출처 문서 준수, 시크릿 읽기·기록 금지(`.env`·credential을 읽지 않고 산출물에 토큰/키를 옮겨 적지 않는다).
- 프로젝트 컨벤션: `docs/code-convention.md` 등이 있으면 작업 전 읽는다.
- 도메인별 원칙 (비용 가드레일, 보안 규칙, 엣지케이스 처리 등).

## 입력/출력 프로토콜
- **입력**: 읽어야 할 파일 경로 (계획서, 설계 문서).
- **출력**: 산출물 경로 + 형식. 후속 에이전트가 소비할 수 있는 구조로.
- **재호출 시**: 이전 산출물이 존재하면 읽고 개선점을 반영. 사용자 피드백이 주어지면 해당 부분만 수정.

## 팀 통신 프로토콜 (에이전트 팀 모드일 때만)
- **수신**: 누구로부터 무엇을 받는가.
- **발신**: 누구에게 언제 무엇을 보내는가 (예: 모듈 완성 시 QA에게 교차검증 요청).
- **작업 요청**: 공유 작업 목록에서 담당하는 작업 유형.

## 에러 핸들링
- 입력이 모호하면 추측하지 말고 {대상}에게 확인.
- 막히면 리더에게 보고 + 대안 제시 (스킵 시 산출물에 명시).

## 협업
- 다른 팀원과의 상호작용 규칙 (incremental 검증 요청, 이전 산출물 참조 등).
```

## 4. 모델 정책

- **기본: 모델 파라미터를 생략한다.** 에이전트는 세션 모델을 상속하며, 이것이 거의 항상 옳다.
- **오버라이드는 예외다.** 작업 특성이 확실할 때만(예: 대량 단순 분류 → 경량 모델) 지정하고, 정의 파일이나 호출부에 이유를 한 줄 남긴다.
- **특정 모델명을 정의 파일에 박지 않는다.** 모델 세대가 바뀌면 하네스 전체가 구식이 된다. "최고 추론이 필요하다" 같은 요구사항은 적되, 모델명 매핑은 호출 시점에 판단한다.
- **reasoning effort도 같은 원칙이다.** 기본은 세션 상속(생략). 기계적 단계(단순 분류·포맷 변환)만 `low`로 낮추고, 최난도 검증·심판 단계만 상향하며 이유를 한 줄 남긴다.

## 5. QA 에이전트 가이드

코드 생성 하네스에는 QA 에이전트를 포함하는 것이 기본이다.

**필수 사항:**
- **타입은 `general-purpose`** — 검증 스크립트·테스트 실행이 필요하므로 읽기 전용 타입(`Explore`)은 불가.
- **핵심은 경계면 교차검증** — "파일이 존재한다"가 아니라, 생산자(API 응답·쿼리·Edge Function)와 소비자(훅·화면·타입)를 **동시에 읽고 shape을 비교**한다. 경계면 버그(필드명 불일치, null 처리 차이, 단위 차이)가 통합 버그의 대부분이다.
- **incremental QA** — 전체 완성 후 1회가 아니라 **각 모듈 완성 직후** 실행한다. 늦게 발견된 경계면 버그는 수정 범위가 누적되어 비싸진다. developer 정의에 "모듈 완성 시 QA에게 교차검증 요청"을, QA 정의에 "발견 즉시 파일:라인 + 수정 방법으로 회신"을 명시한다.
- **TDD 연동** — QA는 테스트의 존재·의미(load-bearing 여부)·통과를 모두 검증한다. 껍데기 단언(테스트가 있지만 아무것도 검증하지 않음)을 잡아낸다.

**QA 루프 종료 조건:** 재검증 2~3회 초과 시 잔여 이슈를 산출물에 "미해결"로 명시하고 사용자에게 판단을 요청한다 (무한 루프 방지).
