# Guksu Harness — 에이전트 & 스킬 아키텍트

도메인 요청을 **에이전트(누가) + 스킬(어떻게) + 오케스트레이션(언제, 어떤 순서로)**의 세 층으로 분해해 하네스를 구축하는 메타 스킬.

## 핵심 원칙

1. **세 층을 분리한다.** 스킬은 "어떻게", 에이전트는 "누가", 오케스트레이터는 "언제·어떤 순서"만 담는다. 한 파일에 섞이면 재사용이 죽는다.
2. **실행 모드는 작업의 형태가 결정한다.** 제어 흐름을 미리 코드로 쓸 수 있으면 **Workflow**(결정적 오케스트레이션), 진행 중 협상·피드백 루프가 필요하면 **에이전트 팀**, 결과만 받으면 되는 단발 위임이면 **서브 에이전트**. 기본 가정 없이 매번 판별한다 → `references/execution-modes.md`.
3. **모델은 세션 상속이 기본.** 에이전트 호출·정의에 모델을 하드코딩하지 않는다. 오버라이드는 작업 특성이 확실할 때만, 이유 주석과 함께. 이유: 하드코딩된 모델명은 모델 세대가 바뀌는 순간 하네스 전체를 구식으로 만든다.
4. **컨텍스트는 공공재다.** SKILL.md 본문 500줄 이내, 세부는 references/로, 반복 코드는 scripts/로. 항상 로딩되는 것은 name+description뿐이다.
5. **하네스는 진화하는 시스템이다.** 실행 피드백을 에이전트·스킬·CLAUDE.md에 반영하고 변경 이력을 남긴다.

## 절대 규칙 — 생성하는 모든 하네스에 내장한다

생성되는 오케스트레이터·에이전트 정의에 아래 규칙을 명시적으로 포함시킨다:

1. **git 작업은 사용자 전담.** commit·push·branch·merge 등 모든 git 명령을 에이전트가 수행하지 않는다. 종료 시 "커밋은 직접 진행하세요"로 안내만 한다.
2. **코드를 생성하는 하네스는 TDD가 기본.** 인수조건 = 테스트 케이스(Red→Green→Refactor). 종료 기준에 테스트 전체 통과를 포함한다. 외부 인프라(DB·외부 SDK)는 단위 대상이 아니라 모킹/스모크로 분리.
3. **산출물은 파일 기반.** 약속된 경로(`docs/` 또는 `_workspace/`)에 쓰고 읽는다. 중간 산출물은 보존한다(감사 추적). 작업 기록은 공통 워크로그 템플릿(`docs/templates/worklog.md` — 1.개요/2.작업내용/3.주의사항)을 따른다 — 형식이 통일되어야 에이전트 간·세션 간에 서로의 기록을 소비할 수 있다.
4. **단일 출처 문서를 따른다.** 프로젝트에 설계 문서·컨벤션 문서가 있으면 그것이 단일 출처다. 어긋나면 임의 판단하지 말고 사용자에게 확인한다.
5. **QA는 경계면 교차검증 + incremental.** 존재 확인이 아니라 생산자(API/쿼리)↔소비자(훅/화면)를 동시에 읽고 shape을 비교한다. 전체 완성 후 1회가 아니라 모듈 완성 직후마다 실행한다.
6. **시크릿은 읽지도 기록하지도 않는다.** `.env`·credential·키 파일을 읽지 않고, 산출물·로그·보고에 토큰/키/비밀번호를 옮겨 적지 않는다. 산출물은 보존되는 파일이므로(규칙 3) 기록된 시크릿은 곧 유출이다. 설정값이 필요하면 키 이름만 언급한다.

규칙 1(git)·6(시크릿)은 지침으로만 두지 않고 생성하는 하네스의 `.claude/settings.json`에 훅·deny 권한으로 기계적으로 강제한다 → `references/hooks-and-permissions.md` (Phase 2에서 구성).

## 인자 해석 — `/harness <인자>`

슬래시 명령어로 호출되면 인자로 Phase 0 분기를 선결정한다. 인자가 없거나 아래 패턴에 맞지 않으면 Phase 0 감사 결과로 판단한다. 어느 경우든 Phase 0의 감사(파일 확인 + validateHarness)는 생략하지 않는다 — 현황을 모르면 어떤 분기도 안전하지 않다.

| 인자 패턴 | 분기 | 실행 범위 |
|----------|------|----------|
| 구축, 구성, 설계 | 신규 구축 | Phase 0~4 전체 |
| 점검, 감사, 현황, 동기화 | 운영/유지보수 | Phase 0 + 불일치 수정 |
| 추가, 확장, 수정 + {대상} | 기존 확장 | 영향 범위만 Phase 1~3 |
| 해체, 제거, 철거 + {대상?} | 해체 | Phase 0 해체 절차 |
| {자유 서술 도메인 요청} | Phase 0 감사로 판단 | — |

## 워크플로우

### Phase 0: 현황 감사

1. `프로젝트/.claude/agents/`, `프로젝트/.claude/skills/`, `프로젝트/CLAUDE.md`를 읽는다.
2. `scripts/validateHarness.mjs <프로젝트 경로>`를 실행해 구조 결함(frontmatter 누락, 끊긴 references 링크, name-디렉토리 불일치)을 잡는다.
3. 분기를 결정하고 사용자에게 실행 계획을 보고한다:
   - **신규 구축** — 에이전트/스킬이 없음 → Phase 1부터 전체 실행
   - **기존 확장** — 에이전트/스킬 추가·아키텍처 변경 → 영향 범위만 Phase 1~3 부분 실행 (중복 검토 필수)
   - **운영/유지보수** — 점검·동기화·수정 → 감사 결과의 불일치 목록을 하나씩 수정하고, 수정마다 CLAUDE.md 변경 이력 갱신 + 재검증
   - **해체** — 하네스 전체 또는 일부(에이전트·스킬 단위) 제거 → 아래 해체 절차 실행

**해체 절차** — 구축의 역순이며 순서가 중요하다:

1. **범위 확정** — 전체인지 부분인지. 부분이면 오케스트레이터·다른 에이전트 정의·CLAUDE.md에서 해당 항목을 참조하는 곳을 모두 역추적한다.
2. **참조 제거 → 파일 삭제 순서로 진행한다.** 반대로 하면 살아있는 정의가 죽은 파일을 가리키는 dead link가 생긴다. 훅·permissions에 하네스가 추가한 항목(`references/hooks-and-permissions.md` 참조)도 함께 정리한다.
3. **CLAUDE.md 정리** — 전체 해체면 하네스 섹션 삭제, 부분이면 변경 이력에 제거 사실과 사유를 기록.
4. **산출물(`docs/`·`_workspace/`)은 감사 추적 자료다** — 임의 삭제하지 않고 보존/삭제를 사용자에게 확인한다.
5. **재검증** — validateHarness error 0건 + 남은 스킬 description 간 트리거 충돌 재점검.

### Phase 1: 설계

1. **도메인 분석** — 핵심 작업 유형(생성·검증·편집·분석), 기술 스택, 데이터 모델, 단일 출처 문서 파악. 사용자의 숙련도를 대화 맥락에서 감지해 커뮤니케이션 톤을 맞춘다.
2. **실행 모드 선택** — 핵심 판별 질문: *"이 작업의 제어 흐름을 지금 코드로 쓸 수 있는가?"*

   | 모드 | 선택 기준 | 조율 수단 |
   |------|----------|----------|
   | **Workflow** | 흐름이 사전 결정 가능(팬아웃·파이프라인·반복), 대량 항목, 스키마 검증된 산출물 | `Workflow` 도구 스크립트 |
   | **에이전트 팀** | 진행 중 협상·피드백 루프(기획↔구현↔QA), 중간 산출물 상호 참조 | `Agent(name)` 스폰+`SendMessage`+`TaskCreate` (암시적 단일 팀) |
   | **서브 에이전트** | 단발 위임, 결과만 회수 | `Agent` 도구 |
   | **하이브리드** | Phase별 특성이 다름 | Phase마다 위 모드 조합 |

   상세 기준·패턴·예시: `references/execution-modes.md` (이 단계에서 반드시 읽는다)
3. **에이전트 분리** — 전문성·병렬성·컨텍스트·재사용성 4축으로 판단. 기존 에이전트/스킬과의 중복을 먼저 검토한다 → `references/agent-design.md`

### Phase 2: 구축

1. **에이전트 정의** — `프로젝트/.claude/agents/{name}.md`. 빌트인 타입(general-purpose 등)을 쓰더라도 정의 파일은 만든다(다음 세션 재사용·협업 프로토콜 명시를 위해). 필수 섹션과 템플릿: `references/agent-design.md`
2. **스킬 생성** — `프로젝트/.claude/skills/{name}/SKILL.md`. description은 적극적(pushy)으로, 본문은 Why 중심·명령형·500줄 이내로 → `references/skill-authoring.md`
3. **오케스트레이터** — 실행 모드별 골격·데이터 전달 프로토콜·에러 핸들링·후속 작업(부분 재실행) 지원: `references/orchestrator-template.md`. 기존 확장이면 새로 만들지 말고 기존 오케스트레이터를 수정한다.
4. **훅·권한 구성** — 절대 규칙 1(git)·6(시크릿)을 기계적으로 강제한다: 이 스킬의 `assets/hooks/` 훅 2종(git 변경 차단 + Bash 경유 시크릿 접근 차단)을 프로젝트 `.claude/hooks/`로 복사해 PreToolUse에 등록하고, permissions deny(Read 도구 측)와 함께 2중 방어를 구성한다. 에이전트가 반복 실행할 테스트·빌드 명령은 allowlist로 사전 허용해 자율 실행이 권한 프롬프트에 끊기지 않게 한다. 코드 생성 하네스면 TDD 종료 게이트(Stop 훅)를 사용자 확인 후 선택 적용 → `references/hooks-and-permissions.md`
5. **공통 템플릿 배포** — 절대 규칙 3의 기록 형식을 구체화한다: 이 플러그인 `docs` 스킬의 템플릿 3종(이 스킬 기준 `../docs/assets/templates/`의 `worklog.md`·`retro.md`·`handoff.md`)을 프로젝트 `docs/templates/`로 복사한다(이미 있으면 보존 — 프로젝트 사본이 단일 출처). 훅과 같은 이유로 베껴 쓰지 않고 그대로 복사한다. 에이전트 정의에 "작업 완료 시 워크로그 기록"을, 오케스트레이터 종료 절차에 "회고 제안(`retro` 스킬)"을, 에러 핸들링에 "세션 중단 시 인계 문서 작성(`handoff` 스킬)"을 명시한다. 절차 상세는 `docs`·`retro`·`handoff` 스킬 참조.

### Phase 3: 검증

1. **구조 검증** — `node {이 스킬 경로}/scripts/validateHarness.mjs <프로젝트 경로>` 실행, error 0건 확인.
2. **트리거 검증** — should-trigger / should-NOT-trigger 쿼리(near-miss 중심)로 description을 점검.
3. **실행 테스트** — 현실적 테스트 프롬프트 2~3개로 실제 실행, 가능하면 with/without-skill 비교.
   방법론 상세: `references/testing-guide.md`
4. 오케스트레이터에 `## 테스트 시나리오` 섹션(정상 1 + 에러 1 이상)이 있는지 확인한다.

### Phase 4: 등록과 진화

1. **CLAUDE.md 포인터 등록** — 트리거 규칙 + 변경 이력 테이블만. 에이전트/스킬 목록·디렉토리 구조는 넣지 않는다(파일 시스템과 오케스트레이터가 단일 출처이므로 중복).

   ````markdown
   ## 하네스: {도메인명}

   **목표:** {한 줄}

   **트리거:** {도메인} 관련 작업 요청 시 `{orchestrator-skill}` 스킬을 사용하라. 단순 질문은 직접 응답 가능.

   **변경 이력:**
   | 날짜 | 변경 내용 | 대상 | 사유 |
   |------|----------|------|------|
   | {YYYY-MM-DD} | 초기 구성 | 전체 | - |
   ````

2. **실행 후 피드백 수집** — 매 실행 완료 후 개선점을 묻는다(강요하지 않되 기회는 제공). 산출물 근거의 체계적 회고는 `retro` 스킬로 실행한다(분석 → 제안 → 승인 → 적용).
3. **피드백 반영 경로** — 결과물 품질→해당 스킬, 역할 문제→에이전트 정의, 순서 문제→오케스트레이터, 트리거 누락→description. 모든 변경은 변경 이력에 기록한다. 같은 피드백이 2회 반복되거나 에이전트가 반복 실패하면 사용자가 요청하지 않아도 진화(`retro` 스킬 실행)를 제안한다.
