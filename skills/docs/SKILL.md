# Docs — 공통 워크로그 기록

모든 작업 기록을 하나의 공통 템플릿으로 남기는 스킬. **형식 통일이 목적이다** — 에이전트마다 기록 형식이 다르면 다른 에이전트·다음 세션·사용자가 기록을 예측 가능하게 소비할 수 없다. 템플릿은 기록자와 소비자 사이의 계약이다.

## 템플릿 — 단일 출처 규칙

이 스킬은 공통 템플릿 5종을 `assets/templates/`에 번들한다:

| 템플릿 | 용도 | 기록 위치 |
|--------|------|----------|
| `worklog.md` | 작업 기록 (이 스킬의 기록 절차가 사용) | `docs/worklog/` |
| `retro.md` | 하네스 회고 (`retro` 스킬이 사용) | `docs/retro/` |
| `handoff.md` | 세션 인계 (`handoff` 스킬이 사용) | `docs/handoff/` |
| `loop-spec.md` | 루프 명세 (`loop` 스킬이 사용) | `docs/loops/` |
| `digest.md` | 지식 다이제스트 (`digest` 스킬이 사용) | `docs/digests/` |

1. **프로젝트에 `docs/templates/{템플릿명}.md`가 있으면 그것이 단일 출처다.** 그대로 사용한다.
2. 없으면 이 스킬의 `assets/templates/{템플릿명}.md`를 프로젝트 `docs/templates/`로 **복사한 뒤** 사용한다. 베껴 쓰지 않고 복사한다 — 복사가 결정적이어야 프로젝트마다 미묘하게 다른 사본이 생기지 않는다.

## 기록 절차

1. **작업 단위 확정** — 하나의 요청/기능/수정이 한 문서다. 너무 잘게 쪼개지 않는다.
2. **경로** — `docs/worklog/{YYYY-MM-DD}-{slug}.md`. 여러 에이전트가 같은 작업에서 병렬로 기록하면 `{YYYY-MM-DD}-{slug}-{agent}.md`로 각자 파일을 만든다 (동시 쓰기 충돌 방지).
3. **채우기** — 템플릿을 복사해 `{중괄호}`를 채우고 안내 주석을 삭제한다.
   - 섹션 구조(1. 개요 / 2. 작업내용 / 3. 주의사항)는 추가·삭제·개명하지 않는다. 세부가 더 필요하면 섹션 안에서 목록·소제목으로 해결한다.
   - 주의사항이 없으면 비워두지 말고 "없음"으로 명시한다 — 빈 섹션은 "안 쓴 것"과 "없는 것"을 구분할 수 없다.
4. **시크릿 금지** — 토큰·키·비밀번호를 옮겨 적지 않는다. 워크로그는 보존되는 파일이므로 기록된 시크릿은 곧 유출이다. 설정값이 필요하면 키 이름만 적는다.

## 후속 갱신

- 같은 작업 단위의 문서가 이미 있으면 새 문서를 만들지 않고 기존 문서를 갱신한다.
- 갱신 시 기존 내용을 지우지 않는다 — "2. 작업내용"에 항목을 추가하고, 방향이 바뀌었으면 바뀐 사실 자체를 기록한다 (감사 추적).
- 사용자가 특정 섹션만 수정을 요청하면 해당 섹션만 고친다.

## 하네스 연동

guksu-harness가 구축한 하네스에서는 Phase 2에서 템플릿 5종이 프로젝트 `docs/templates/`로 복사되고, 각 에이전트 정의에 "작업 완료 시 워크로그 기록"이 명시된다. 기록은 오케스트레이터가 아니라 **작업한 에이전트 본인이 남긴다** — 내용을 아는 쪽이 기록해야 정확하다. 회고·인계·루프·다이제스트 기록의 절차는 각각 `retro`·`handoff`·`loop`·`digest` 스킬이 담당하며, 이 스킬은 템플릿의 단일 출처만 관리한다.
