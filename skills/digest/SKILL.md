# Digest — 세션 간 지식 캐시

프롬프트 캐시는 세션 안에서만 산다. 세션이 바뀌면 같은 대형 파일을 처음부터 다시 읽는다 — 이 스킬은 분석 결과를 파일로 옮겨 그 반복 비용을 없앤다. `handoff`가 "작업 상태"의 캐시라면 digest는 "코드 이해"의 캐시다.

## 원칙

- **다이제스트는 지도이지 원문 대체가 아니다.** 수정할 파일은 반드시 원문을 읽는다 — 요약만 믿고 편집하면 요약의 공백이 버그가 된다. 다이제스트가 답하지 못하는 질문이 나오면 원문으로 내려간다.
- **신선도는 내용 해시로 판정한다.** 소스 파일의 해시를 frontmatter에 기록하고 사용 전 `scripts/checkFreshness.mjs`로 검증한다. mtime은 체크아웃·복사만으로도 바뀌므로 쓰지 않는다. 틀린 캐시는 없는 캐시보다 나쁘다 — 검증 없이 소비하지 않는다.
- **시크릿은 기록하지 않는다.** 다이제스트는 보존되는 파일이다. 설정값이 필요하면 키 이름만 적는다.

## 작성·갱신

1. **단위 확정** — 소비자 관점에서 한 번에 읽힐 단위(모듈·서브시스템·대형 파일 1개)로 잡는다. 프로젝트 전체를 한 문서에 넣지 않는다 — 클수록 stale이 잦아지고 갱신이 비싸진다.
2. **템플릿** — 프로젝트 `docs/templates/digest.md`가 단일 출처. 없으면 docs 스킬의 절차로 복사한다.
3. **경로** — `docs/digests/{slug}.md`. slug는 대상 이름(모듈·경로 기반, 날짜 아님 — 갱신형 문서다). 같은 대상의 문서가 있으면 새로 만들지 않고 갱신한다.
4. **해시 기록** — `node {이 스킬 경로}/scripts/checkFreshness.mjs hash <소스 파일...>`로 소스별 해시를 얻어 frontmatter `sources`에 기록한다. 손으로 적거나 추측하지 않는다.
5. **내용** — 다음 소비자가 원문 없이 착수할 수 있는 것만 담는다: 책임, 공개 인터페이스(시그니처·shape), 의존 방향, 불변식과 함정. 코드 전문을 옮겨 적지 않는다 — 그럴 거면 원문을 읽는 편이 낫다.

## 사용(소비)

1. 분석·구현 착수 전 `docs/digests/`에서 대상 다이제스트를 찾는다.
2. `node {이 스킬 경로}/scripts/checkFreshness.mjs check <다이제스트 경로> --root <프로젝트 루트>`를 실행한다:
   - **fresh (전부)** — 다이제스트만 읽고 진행한다. 원문은 수정 대상 파일만 연다.
   - **stale** — 바뀐 소스만 다시 읽고 다이제스트를 갱신(해시 포함)한 뒤 진행한다. 전체 재작성이 아니라 바뀐 부분만 고친다.
   - **missing** — 소스가 삭제/이동됐다. 다이제스트에서 해당 항목을 정리하거나, 문서 자체의 폐기를 사용자에게 확인한다.
3. 작업 중 다이제스트와 원문의 불일치를 발견하면 그 자리에서 다이제스트를 현실에 맞게 고친다.

## 하네스 연동

guksu-harness가 생성하는 하네스에서는 Phase 2에서 digest 템플릿이 `docs/templates/`로 복사되고, 리서치·분석을 맡는 에이전트 정의에 "착수 전 `docs/digests/` 확인, 대형 분석 완료 시 다이제스트 기록"이 명시된다 — 절대 규칙 7(컨텍스트 절약형 설계)의 세션 간 실행 수단이다. 독립적으로도 언제든 호출 가능하다.
