# Loop — 루프 설계자

루프는 "실행 → 검증 → 계속/중지 판정"의 반복이다. 프롬프트가 아니라 루프를 설계하는 것이 요점이다 — 사람이 매 반복 결과를 확인하는 대신, **검증자와 종료 규칙이 그 판단을 대신한다.** 따라서 루프의 품질은 검증자의 품질이고, 검증할 수 없는 루프는 만들지 않는다.

## 절대 원칙

1. **자기평가 금지.** 종료 판정은 기계적 검증(테스트·타입체크·린트·빌드 등 실행 명령의 종료 코드)만 쓴다. "내가 보기에 다 됐다"는 판정이 아니다. 검증 수단은 신뢰도 순으로 규칙 기반(종료 코드) > 시각·실행 피드백 > LLM 판정이며, **LLM 판정은 종료 규칙으로 쓰지 않는다** — Anthropic이 robust하지 않다고 명시했고, 자기평가 금지가 곧 이 뜻이다(agent-design QA 가이드의 검증 수단 서열과 동일).
2. **4요소는 사용자 확인 없이 실행하지 않는다.** 트리거·실행 단위·검증자·종료 규칙을 제안값과 함께 제시하고 확인받는다 (아래 Phase 2).
3. **안전장치 필수 — 토큰 예산 포함.** 최대 반복 횟수와 토큰 예산이 없는 루프는 실행하지 않는다. 예산 초과 시 루프를 계속하지 않고 **자동 중단 → 진행 상황·남은 실패·사유 보고 → 종료**한다.
4. **생성자와 검증자를 분리한다.** 코드를 쓴 에이전트가 자기 결과를 검증하면 같은 맹점을 공유한다 (agent-design과 동일 원칙).
5. git 작업 금지·시크릿 읽기·기록 금지는 여기서도 동일하다.

## 워크플로우

### Phase 1: 적합성 판정

핵심 질문: **"종료를 기계적으로 검증할 수 있는가?"**

- 검증 명령이 존재한다(테스트·빌드·린트 등) → 진행.
- 없다("코드를 더 예쁘게" 같은 주관 목표) → 루프를 만들지 않는다. 검증 가능한 목표로 재정의를 제안하거나(예: "린트 규칙 X 위반 0건"), 1회성 작업으로 전환한다.

### Phase 2: 사용자 확인 — 생략 금지

루프 후보로 판별되어도 임의로 돌리지 않는다. 다음을 제안값과 함께 제시하고 사용자 확인을 받는다 (AskUserQuestion 도구가 있으면 사용):

1. **루프를 사용할지 여부** — 1회 실행으로 충분할 수도 있다.
2. **4요소** — 트리거 / 실행 단위 / 검증자 / 종료 규칙.
3. **안전장치** — 최대 반복(기본 10회), **토큰 예산**(작업 규모에 맞는 제안값), 막힘 판정(같은 에러 3연속 등).

확인 결과는 루프 명세의 "사용자 확인" 필드에 날짜와 함께 기록한다.

### Phase 3: 루프 명세 기록

docs 스킬의 루프 명세 템플릿(`docs/templates/loop-spec.md`, 없으면 docs 스킬 절차로 복사)으로 `docs/loops/{slug}.md`에 기록한다. 명세 없이 루프를 실행하지 않는다 — 무엇을 돌렸고 왜 멈췄는지가 파일로 남아야 감사 추적이 된다(절대 규칙 3).

### Phase 4: 실행 수단 매핑

| 상황 | 실행 수단 |
|------|----------|
| 조건 충족까지 반복 (테스트 통과, 린트 클린) | `/goal` — 평가자가 매 턴 조건 검사 |
| 주기적 반복 (배포 감시, 상태 폴링) | `/loop` — 간격 지정 |
| 턴 종료 시점의 기계적 강제 | **검증자 게이트** (Stop 훅) — 아래 Phase 5 |
| 대량 항목 × 반복 수렴 (스크립트로 제어 가능) | Workflow 반복 패턴 — `while` + `budget.remaining()` 가드 (execution-modes.md 참조) |

수단은 배타적이지 않다 — /goal 루프에 검증자 게이트를 겹치면 종료 규칙이 이중으로 강제된다.

### Phase 5: 검증자 게이트 구성

harness 스킬의 `assets/hooks/verifierGate.mjs`를 프로젝트 `.claude/hooks/`로 복사하고 Stop 훅으로 등록한 뒤, 옆에 `verifierGate.config.json`을 명세와 **일치하게** 생성한다:

```json
{
  "checks": [{ "name": "test", "command": "npm test" }],
  "maxIterations": 10,
  "maxTokens": 500000,
  "stuckAfter": 3
}
```

- `checks` 실패 시 턴 종료가 차단되고 실패 출력이 피드백된다 (종료 규칙 강제).
- `maxTokens`(transcript 누적 토큰)·`maxIterations` 도달 시 루프를 계속하지 않고 "보고 후 종료"가 지시된다 (안전장치 — 토큰 예산 자동 중단).
- `stuckAfter`: 같은 실패 시그니처가 N연속이면 막힘으로 판정해 예산 소진 전에 중단·보고한다. **명세의 "막힘 판정" 값을 이 필드로 옮긴다** — 명세에만 적고 config에 없으면 막힘 판정이 기계적으로 강제되지 않는다.
- 등록 방법·주의사항: harness 스킬의 참조 문서 `hooks-and-permissions.md` §5.

## 실행 중과 종료

- 매 반복의 결과를 명세의 "4. 실행 기록"에 갱신한다 — 진행 로그가 막힘 감지의 근거다.
- 같은 에러가 막힘 판정 횟수만큼 반복되면 반복을 소진하지 말고 중단·보고한다. 중단 시 이어서 작업할 수 있게 handoff 스킬로 인계 문서를 남긴다.
- 종료 시(성공이든 중단이든) 명세의 "5. 종료 보고"를 채우고 상태를 갱신하며, 최종 기록은 docs 스킬의 워크로그로 남긴다.

## 하네스 연동

guksu-harness의 Phase 1 실행 모드 판별에서 반복·수렴형 작업이 감지되면 이 스킬로 위임된다 — 하네스가 임의로 루프를 돌리는 일은 없다 (Phase 2의 사용자 확인이 항상 선행된다).
