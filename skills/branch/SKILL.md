# Branch — 작업 브랜치 확인

보호 브랜치(main 등) 위에서 시작된 작업은 커밋 시점에 문제가 된다 — 그때는 이미 변경이 쌓여 있다. 이 스킬은 **작업 시작 시점**에 브랜치를 확인받아 그 상황을 예방한다. `branchGuard` 훅(보호 브랜치 편집 차단)과 한 쌍이다: 훅이 그물, 스킬이 절차다.

## 원칙

- **확인 없이 전환하지 않는다.** 브랜치 생성·이동은 반드시 사용자 승인 후에 실행한다. 이름을 제안하되 결정은 사용자 몫이다.
- **`git switch`만 쓴다.** 절대 규칙 1(git 사용자 전담)의 유일한 예외가 "사용자 확인된 브랜치 전환"이며, 수단은 `git switch -c <이름>`(신규)·`git switch <이름>`(기존)으로 한정된다. switch는 로컬 변경과 충돌하면 스스로 거부하므로 작업 내용을 파괴하지 않는다. 파괴·이탈 플래그(`-f`·`--discard-changes`·`-C`·`--orphan`·`-d`/`--detach`)와 `checkout`은 훅이 계속 차단한다 — detached HEAD는 branchGuard를 무력화하므로 전환 대상은 항상 이름 있는 브랜치다.
- **커밋·푸시·병합·브랜치 삭제는 여전히 사용자 전담이다.** 이 스킬은 "어디서 작업할지"만 다룬다.

## 절차

1. **현황 파악** — `git branch --show-current`와 `git status -sb`로 현재 브랜치·미커밋 변경을 확인한다. 보호 브랜치 목록은 `.claude/hooks/branchGuard.config.json`의 `protectedBranches`가 단일 출처다(없으면 main·master).
2. **판정**:
   - 보호 브랜치가 아니면 → 브랜치 이름이 이번 작업과 맞는지만 짧게 확인하고 진행한다(무관한 이전 작업 브랜치 위일 수 있다).
   - 보호 브랜치면 → 파일 편집 전에 반드시 3에서 확인받는다.
3. **사용자 확인** — 선택지를 제시한다:
   - **새 브랜치 생성(기본 제안)** — 이름을 제안한다: 프로젝트의 기존 브랜치 이름 패턴(`git branch -a`)을 먼저 따르고, 없으면 `{type}/{slug}`(feat/fix/chore + 작업 요약)로. 승인 시 `git switch -c <이름>`.
   - **기존 브랜치로 이동** — 관련 작업 브랜치가 이미 있으면 후보로 제시. 승인 시 `git switch <이름>`.
   - **보호 브랜치에서 계속** — 에이전트가 대신 우회하지 않는다. 사용자가 직접 `branchGuard.config.json`에서 해당 브랜치를 빼야 하며, 그 방법만 안내한다(의도된 마찰 — 예외는 명시적이어야 한다).
4. **전환 실행** — 승인된 명령을 실행하고 `git branch --show-current`로 결과를 확인한 뒤 작업을 시작한다.
5. **전환이 거부되면** — 미커밋 변경 충돌로 git이 switch를 거부한 경우, 변경 목록을 보고하고 처분(커밋·stash)은 사용자에게 맡긴다 — stash도 git 변경 명령이므로 에이전트가 하지 않는다.

## 하네스 연동

guksu-harness가 생성하는 하네스에서는 `branchGuard.mjs` 훅이 Phase 2에서 `.claude/hooks/`로 복사되어 보호 브랜치 편집을 기계적으로 차단하고, 오케스트레이터 Phase 0(컨텍스트 확인)에 "작업 브랜치 확인"이 첫 단계로 명시된다. 독립적으로도 언제든 호출 가능하다.
