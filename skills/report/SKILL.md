# Report — 작업 종료 HTML 보고서

작업 종료 보고를 채팅 텍스트가 아니라 **파일로 남는 HTML 문서**로 전환한다. 채팅 보고는 스크롤에 묻히고 세션이 끝나면 사라진다 — 보고서는 `docs/reports/`에 날짜순으로 누적되어 스프린트 문서처럼 히스토리가 된다. 워크로그(md)가 **정본 기록**이고, 이 보고서는 요약 + **사용자 검토 필요** + **후속 조치**가 추가된 **사용자 검토용 뷰**다 — 보고서가 워크로그를 대체하지 않는다.

## 원칙

- **채팅 보고는 요약으로 축소한다.** 보고서 생성 후 채팅에는 3~5문장 요약 + 보고서 경로 + (있다면) 검토 필요 항목 수만 남긴다. 상세는 전부 보고서로.
- **검토·후속 섹션은 판단의 산물이다.** "사용자 검토 필요"에는 사용자의 결정·승인이 필요한 것만(선택지와 영향 병기), "후속 조치"에는 트리거·담당이 명시된 실행 가능한 항목만 담는다. 없으면 "없음"을 명시한다 — 빈 섹션은 "안 쓴 것"과 구분되지 않는다.
- **형식은 템플릿이 계약이다.** 섹션 구조(1.요약 / 2.작업 내용 / 3.검증 결과 / 4.사용자 검토 필요 / 5.후속 조치)와 스타일을 임의 변경하지 않는다 — 보고서 간 형식이 통일되어야 히스토리로 소비된다. 테마는 중립(백지·슬레이트·표준 상태색)으로 고정되어 있다.
- **시크릿 금지.** 보고서는 보존·공유되는 파일이다.

## 생성 절차

1. **템플릿 확정** — 프로젝트 `docs/templates/report.html`이 단일 출처. 없으면 docs 스킬의 절차로 복사한다.
2. **경로** — `docs/reports/{YYYY-MM-DD}-{slug}.html`. slug는 워크로그와 동일하게 맞춘다(같은 작업 단위 = 같은 slug — 워크로그↔보고서가 쌍으로 추적된다). 같은 작업의 보고서가 이미 있으면 새로 만들지 않고 갱신한다.
3. **채우기** — `{중괄호}`를 채우고 안내 주석을 삭제한다:
   - **1. 요약**: 무엇을·왜·어떤 상태로 끝났는지 3~5문장.
   - **2. 작업 내용**: 변경 표(구분/경로/내용) — 워크로그의 "2. 작업내용"과 일치해야 한다.
   - **3. 검증 결과**: 실행한 검증(테스트·타입체크·실행 확인)과 실제 결과. 실패가 있으면 숨기지 않고 `fail`로 표기.
   - **4. 사용자 검토 필요**: 결정 대기 항목 — 왜 사용자 판단인지, 선택지, 미결정 시 기본 동작.
   - **5. 후속 조치**: 다음 단계·미해결 이슈 — 트리거(언제)와 담당(누가) 명시.
4. **전달** — 보고서 파일을 사용자에게 전달한다(렌더 가능한 환경이면 파일 전송, 아니면 경로 안내). 채팅에는 요약만(원칙 1).

## 히스토리

- 보고서는 삭제·덮어쓰기하지 않고 누적한다(감사 추적 — 절대 규칙 3). 파일명 날짜 접두사가 시간순 정렬을 보장하므로 별도 인덱스 파일은 만들지 않는다 — 파일 시스템이 단일 출처다.
- 방향이 뒤집힌 재작업은 새 보고서를 만들고, 이전 보고서의 "후속 조치"에서 이어졌음을 요약에 한 줄로 밝힌다.

## 하네스 연동

guksu-harness가 생성하는 하네스에서는 Phase 2에서 report 템플릿이 `docs/templates/`로 복사되고, 오케스트레이터 종료 절차가 "HTML 보고서 생성(report 스킬) → 채팅에는 요약·경로·검토 항목 수만"으로 명시된다. 검토 필요 항목이 승인되면 그 결과는 CLAUDE.md 변경 이력 또는 워크로그에 기록한다. 독립적으로도 언제든 호출 가능하다.
