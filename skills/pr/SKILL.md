# PR — 커밋·PR 업로드 (사용자 명시 요청 시)

절대 규칙 1("git 작업은 사용자 전담")의 **두 번째 예외**다. 기본값은 여전히 사용자 전담이며, 이 스킬은 사용자가 커밋·PR 업로드를 **명시적으로 요청했을 때만** 발동한다. `blockGitMutation` 훅의 `allowCommitPush` 옵트인과 한 쌍이다: 훅이 그물, 스킬이 절차다.

## 원칙

- **명시 요청 없이는 발동하지 않는다.** 작업이 끝났다고 알아서 커밋하지 않는다. "이제 끝났나?" 같은 애매한 발화는 요청이 아니다 — 커밋할지 물어본다.
- **커밋 메시지에 Claude 작성 표기를 절대 넣지 않는다.** `Co-Authored-By: Claude ...`, `🤖 Generated with [Claude Code](...)`, `Claude-Session:` 링크, `noreply@anthropic.com` 등 어떤 형태든 금지다. PR 제목·본문도 동일하다(모델명 포함). 훅이 표기가 든 커밋을 기계적으로 차단하지만, 훅이 없는 환경에서도 스스로 지킨다 — 커밋 직전 메시지 전문을 훑어 표기·시크릿(절대 규칙 6)이 없는지 최종 확인한다.
- **git-flow는 main ← dev ← feat이다.** 작업은 `feat/{slug}` 브랜치에서 하고, PR 베이스는 `dev`다. `dev → main`(릴리스)은 사용자 전담 — 에이전트는 요청받지 않는 한 만들지 않는다. 저장소에 `dev`가 없으면 기본 브랜치를 베이스로 쓰되 사용자에게 알린다.
- **범위는 add → commit → push → PR 생성까지다.** merge·rebase·reset·force push·원격 브랜치 삭제는 예외 없이 사용자 전담이며 훅이 계속 차단한다. **PR 머지도 하지 않는다** — 리뷰와 머지는 사용자 몫이다.

## 커밋 메시지 규칙

- **Conventional Commits**: `{type}: {요약}` (type: feat·fix·docs·refactor·test·chore). 본문은 필요할 때만 — 무엇을·왜(어떻게는 diff가 말한다).
- 언어·상세도는 프로젝트의 기존 커밋 로그를 따른다 (`git log --oneline -20`으로 확인).
- 메시지는 **`-m` 인라인으로만** 작성한다(여러 줄은 heredoc). `-F`/`-t`/`-c`/`-C`/`--amend`/`--fixup`은 훅이 차단한다 — 메시지를 검사할 수 없거나 히스토리를 재작성하기 때문이다. 메시지를 고치려면 새 커밋이 아니라 사용자에게 안내한다(amend는 사용자 전담).
- 논리 단위로 나눈다. `git add .`가 아니라 **경로를 명시해** 스테이징한다 — 의도치 않은 파일(시크릿·로컬 설정)이 섞이는 것을 막는다.

## 절차

1. **요청 확인** — 사용자의 명시 요청인지 확인한다. 하네스 오케스트레이터가 대신 호출한 경우에도 최초 근거는 사용자 발화여야 한다.
2. **브랜치 확인** — `git branch --show-current`. `feat/*` 위가 아니면(main·dev 등) `branch` 스킬로 사용자 확인 후 `git switch -c feat/<slug> dev`로 이동부터 한다.
3. **변경 검토** — `git status -sb`·`git diff`로 커밋 대상을 확정하고 논리 단위로 나눈다. 시크릿·의도치 않은 파일은 제외한다.
4. **커밋** — 메시지를 확정하고(여러 커밋이면 계획을 먼저 짧게 보고) `git add <경로>` + `git commit -m`으로 실행한다. 커밋 직전 메시지 전문에서 Claude 표기·시크릿 부재를 확인한다.
5. **푸시** — `git push -u origin <feat-브랜치>`. force 계열은 쓰지 않는다(훅 차단). 네트워크 실패면 지수 백오프로 최대 4회 재시도한다.
6. **PR 생성** — 베이스는 `dev`. 제목은 대표 커밋과 같은 규칙, 본문은 변경 요약·검증 결과(테스트 등)·리뷰 포인트를 담는다. 저장소에 PR 템플릿이 있으면 그 구조를 따른다. 본문에도 Claude 표기·모델명을 넣지 않는다. 수단은 환경이 제공하는 것을 쓴다(`gh pr create` 또는 GitHub MCP 도구).
7. **보고** — PR URL과 커밋 목록을 보고하고 머지는 사용자에게 맡긴다.

## 하네스 연동

이 예외는 **옵트인**이다. 하네스 구축(Phase 2) 시 사용자에게 PR 플로우 활성화 여부를 확인받고, 승인 시에만:

- `.claude/hooks/blockGitMutation.config.json`에 `{ "allowCommitPush": true }`를 쓴다. config가 없으면 훅은 commit·push를 계속 차단한다(기본값).
- git-flow 채택 시 `branchGuard.config.json`의 `protectedBranches`에 `dev`(또는 `develop`)를 추가한다 — 작업은 항상 `feat/*`에서.

세션 중 훅이 커밋을 차단했는데 사용자가 방금 명시 요청한 상황이면, 옵트인 활성화 여부를 사용자에게 확인받은 뒤 config를 갱신하고 CLAUDE.md 변경 이력에 기록한다 — 사용자 요청 없이 에이전트가 스스로 config를 켜는 것은 금지다.
